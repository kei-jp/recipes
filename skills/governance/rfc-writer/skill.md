# このスキルを利用するタイミング

以下の場合に利用すること。

- 設計検討を記録する
- 未確定事項を整理する
- 疑義を管理する
- 要件整理を行う
- 意思決定前の論点を整理する
- ユーザー確認事項を管理する
- 将来的な ADR 候補を蓄積する

---

# 基本方針

- この Skill は、まだ決定していない内容を扱うこと。
- 確定済み事項を RFC に記載してはいけない。
- 設計者として結論を先取りしてはいけない。
- レビューや議論の論点を整理することに集中すること。
- 推測を事実として書いてはいけない。
- 未確定事項は未確定のまま残すこと。
- 変更履歴と判断の経緯を残すこと。

---

# RFC の対象

以下を RFC の主な対象とすること。

- Open Question
- Design Discussion
- Requirement Discussion
- Trade-off Analysis
- User Decision Pending

---

# 保存先

- RFC は原則として `docs/rfc/` に保存すること。
- ディレクトリが存在しない場合は作成すること。
- 進行中の RFC と完了済みの RFC を同じ場所で混在させてもよいが、必要に応じてサブディレクトリで分けること。
- 既存 RFC を上書きせず、時系列で追加すること。

---

# ファイル命名規則

- ファイル名は次の形式とすること。

```text
YYYY-MM-DD_HHmmss_<project-or-topic>_<summary>.md
```

- ファイル名には以下が分かるようにすること。
  - 作成日時
  - 対象
  - 要約
- 同一日付内で重複しないように時刻を含めること。

### 例

```text
2026-07-01_185120_listing_policy_worker_hs_code_review.md
```

---

# RFC テンプレート

RFC は以下の構造を標準とすること。

```text
# タイトル

## 背景

## 議題

### <項目名>

#### 論点

#### 現時点の整理

#### 選択肢

#### Decision Candidate

#### 未確定

#### ユーザー回答欄
```

- 見出し構造を使用すること。
- 番号付きカウンタで管理しないこと。
- 時系列を維持すること。
- 過去 RFC を上書きしないこと。

---

# 記載ルール

- 未確定事項を明示すること。
- 推測を事実として書かないこと。
- 議論中の論点と確定事項を混在させないこと。
- 変更履歴を残すこと。
- 必要に応じて選択肢を複数示すこと。
- Decision Candidate は、確定前の有力候補として整理すること。
- ユーザー回答欄は、確認待ちの内容を明確にすること。

---

# ADR との関係

- RFC は議論と整理を目的とすること。
- ADR は決定事項を記録すること。
- 重要な設計判断が確定した場合は ADR 作成候補として扱うこと。
- RFC から ADR へ昇格できる粒度で整理すること。
- RFC を ADR の代替にしないこと。

# このスキルを利用するタイミング

以下の場合に利用すること。

- GitHub Project を新規作成する
- GitHub Project を更新する
- GitHub Project の項目を追加する
- GitHub Project の項目を移動する
- GitHub Project の項目を削除する
- GitHub CLI (`gh`) で Project を操作する

---

# 基本方針

- Project は、作業の進捗や優先度を可視化するために使うこと。
- Issue の本文整備や Close はこの Skill で扱わないこと。
- Issue の作成や更新は、必要であれば別 Skill を使うこと。
- Project は「見える化」と「並び替え」を主目的にすること。
- Project では状態管理を `Status`、優先度管理を `Priority` で行うこと。
- `Area` は必要な場合のみ付与し、ラベルとの重複管理を避けること。

---

# 判断フロー

1. 対象が Project 自体の管理か確認すること。
2. 既存の Project に対する更新か、新規作成か判断すること。
3. 項目を扱う場合は、追加・移動・削除のどれかを決めること。
4. 必要な項目が Issue か Pull Request かを確認すること。
5. 変更前に既存の Project 構成を確認すること。
6. 判断に迷う場合は、勝手に変更せずユーザーへ確認すること。

---

# Project を作成・更新する条件

以下に該当する場合は Project を作成または更新してよいこと。

- 依頼対象の作業を一覧化したい
- 進捗管理を見える化したい
- 既存の並び順や状態を整理したい
- 複数の Issue や Pull Request をまとめて追いたい
- 作業の優先度や状態を反映したい

---

# Project を変更しない条件

以下は Project 化しないこと。

- 一時的な試行錯誤だけで終わる内容
- 後から追跡する必要がない内容
- 依頼範囲に関係しない補助的なメモ
- 変更理由を説明できない並び替え

---

# 変更前確認

- 変更前に既存 Project の名前、対象、列またはフィールドの構成を確認すること。
- 項目を移動する前に、対象アイテムの現在位置を確認すること。
- 項目を追加する前に、対象アイテムが存在するか確認すること。
- 重複追加を避けること。

---

# フィールド方針

## Status

- `Status` は必須フィールドとして扱うこと。
- 状態は次のいずれかで運用すること。
  - `Backlog`
  - `Ready`
  - `In Progress`
  - `Blocked`
  - `Review`
  - `Done`
- 基本の遷移は `Backlog -> Ready -> In Progress -> Review -> Done` とすること。
- `Blocked` はどの状態からでも入れられる例外状態として扱うこと。
- 状態を変更する前に、変更理由を確認すること。

## Priority

- `Priority` は可能な限り付与すること。
- 優先度は次のいずれかで運用すること。
  - `P0`
  - `P1`
  - `P2`
  - `P3`
- 緊急性や重要度が変わった場合のみ更新すること。
- 変更理由を説明できない優先度変更は行わないこと。

## Area

- `Area` は対象が明確な場合のみ付与すること。
- `Area` は `area/` プレフィックスで統一すること。
- 対象がプロジェクト名やアプリ名なら、その名前をそのまま使ってよいこと。
- 例として次を許容すること。
  - `area/backend`
  - `area/frontend`
  - `area/sql`
  - `area/infra`
  - `area/docs`
  - `area/listing_policy_worker`
  - `area/wasabiswitch`

---

# 項目管理

- 項目は Issue または Pull Request を基本に扱うこと。
- 必要に応じて Project に関連するメモや作業単位を追加すること。
- 項目を移動する場合は、状態や優先度の変化に応じて行うこと。
- 項目を削除する場合は、不要になった根拠を確認すること。
- 同じ内容の項目を複数追加しないこと。
- 項目の状態は `Status` に、優先度は `Priority` に反映すること。
- `Area` は分類が必要なときだけ補助的に使うこと。

---

# `gh` 利用

- `gh` が利用可能な環境では CLI による操作を優先してよいこと。
- `gh` を使う場合は、先に対象 Project の現在状態を確認すること。
- 使う操作は Project の作成、更新、項目追加、項目移動、項目削除に限定すること。
- 必要なら Issue や Pull Request の参照情報を確認してから操作すること。

---

# 出力方針

- Project の変更内容は簡潔かつ具体的に記載すること。
- 何を、どの項目に対して、どう変えたかが分かるようにすること。
- 変更理由がある場合は一緒に示すこと。

---

# 自律的な判断

- Project を更新できる状況であれば、毎回ユーザー確認を求めないこと。
- ただし以下の場合は確認すること。
  - 対象 Project が曖昧
  - 項目の追加先が曖昧
  - 列やフィールドの変更影響が大きい
  - 重複追加の可能性が高い
